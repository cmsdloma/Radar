01 · Overview

What is Radar?

Radar is a peer-to-peer Android application for local device discovery and off-grid communication. It enables you to discover nearby devices, visualize them on a radar or map, and message them directly without internet connectivity or central servers.

Core purpose

  • Discover nearby Android devices running Radar
  • See where peers are relative to your position
  • Chat directly via local transports
  • Discover all devices on local WiFi networks

Key differentiators

  • Works fully offline – no cloud required
  • Multi-protocol: BLE, WiFi Aware, mDNS, internet (optional)
  • Transport-agnostic message routing
  • Mesh store-and-forward relaying
  • Real-time location visualizations
  • Privacy-first, local-only by default

02 · Core Features

Visual radar, messaging & mesh networking

Radar combines a live visual radar with a transport-agnostic messaging engine and mesh relay logic to keep conversations flowing even when peers are only intermittently reachable.

Visual Radar Display

Circular radar with distance rings, GPS-based bearing, color-coded peers (male, female, unknown, drone), and real-time updates. Tap any peer to view their profile or jump into chat.

Directional positioning Field-of-view mode Map view integration

Multi-protocol messaging

Direct and broadcast messages are delivered via WiFi Aware, internet relay, or Bluetooth (planned), with automatic retry, acknowledgments, sequence numbers, and delivery state tracking.

Direct & broadcast Acks & read receipts Exponential backoff

Mesh networking

Messages can hop through intermediate peers with TTL limits, loop protection, duplicate prevention, and expiry – enabling store-and-forward messaging in dynamic topologies.

TTL: 5 hops Loop prevention Forwarded: direct & broadcast

Profile & SMS integration

Rich profiles with avatars, demographics, interests, tags, and device info – plus virtual profiles for SMS contacts using deterministic UUIDs.

Automatic exchange Photo transfer Virtual SMS profiles
03 · Device Discovery Protocols

BLE, WiFi Aware, mDNS & network scanning

Radar layers multiple discovery mechanisms so that peers can find each other using whatever radios and network connectivity are available.

Bluetooth Low Energy (BLE)

Custom advertising with UUID, coarse location, age, and gender; background scanning with RSSI-based distance estimation and MAC-based vendor detection.

  • Custom Bluetooth service UUID
  • Custom manufacturer ID
  • Update frequency: 10s background · 2s foreground
  • Battery impact: ~2–5% per hour

WiFi Aware (NAN)

Primary discovery and messaging transport using publish/subscribe and PSK-secured data paths for reliable JSON-based communication.

  • Custom Radar service name
  • Shared passphrase (API 29+)
  • Port range: 5000–5100
  • Foreground-only due to OS limitations

mDNS & network segment scanning

mDNS discovers Radar peers on the LAN, while ARP cache, ping sweeps, and reverse DNS enumerate all devices on the subnet.

  • Custom mDNS service type
  • ARP + ping sweep (20 concurrent threads)
  • Typical scan: 192.168.x.1–254 in ~10–15s
  • Smart merging of duplicate sightings

Internet-based discovery (optional)

When enabled, locations and messages can be relayed via an Oracle APEX backend with ORDS APIs and FCM push notifications for global reach.

Location history Cloud message relay FCM notifications

Protocol snapshot

Protocol Role Range
BLE Low-power discovery & handshake ~10–100 m
WiFi Aware Primary discovery & messaging ~10–100 m
mDNS / LAN WiFi peers & network devices Local subnet
Internet Global reach & fallback Worldwide

Device support and Android background restrictions are handled internally so the app can gracefully fall back to whatever transports are available.

04 · Messaging System

Transport-agnostic delivery with acks & retries

The messaging layer separates message delivery logic from transport implementation. A central delivery service continuously scans for pending messages, selects the best available route, and tracks state through to acknowledgment.

Delivery pipeline

1. User sends message → message record created (status: pending)
2. Delivery service checks every 1 second
3. Attempts WiFi Aware → Internet → Bluetooth
4. On success: pending → sent
5. Recipient auto-sends acknowledgment
6. Sender marks as sent + acknowledged (✓✓) in the UI.

Automatic retry (up to 3 attempts) Exponential backoff 2s / 4s / 8s In-flight tracking & duplicate prevention

Message types

  • Direct messages (1:1)
  • Broadcast messages (multi-recipient)
  • Acknowledgments and read receipts
  • Profile exchange messages
  • Chat initialization for deterministic chat IDs

Sequence-based acknowledgments

Each chat has a monotonically increasing sequence number per message. Acks carry just the sequence number, making them compact, ordered, and easy to debug.

  • Used only for direct messages (broadcasts are unacked)
  • Provides implicit ordering within each chat
  • No retry if ack fails – message already delivered
05 · Profile System

Rich profiles & deterministic virtual identities

The profile model describes user identity and device attributes, including avatars, demographics, interests, tags, and MAC/IP info. Profiles are exchanged automatically when you tap peers on the radar or map.

Core & extended fields

  • UUID – 32-char hex identifier
  • Nickname, given name, family name
  • Email, phone number (E.164)
  • Gender (Male, Female, Drone)
  • Date of birth, age, photo
  • Interests, tags, metadata, version
  • Device fields: MAC, IP, manufacturer, last seen

Automatic exchange & photos

When you tap a peer, Radar requests a full profile (if incomplete). The peer responds with a profile response containing a Base64-encoded JPEG avatar, which is decoded and stored in local profile photo storage.

Virtual profiles (SMS)

For non-Radar contacts, virtual profiles are created from SMS data. Phone numbers are normalized to E.164 and mapped deterministically to UUIDs, keeping SMS and Radar messages for the same contact unified under one profile without duplicates.

06 · Network Scanning, Persistence & Services

LAN visibility, saved state & background operation

Radar discovers everything on your LAN, persists chats and profiles across restarts, and keeps advertising your presence via a foreground service when running in the background.

Network segment scanner

Uses ARP cache, ping sweeps, and reverse DNS to identify devices on the local subnet, distinguishing Radar peers from generic devices.

  • 20 concurrent ping threads
  • Scans 192.168.x.1–254 in ~10–15 seconds
  • MAC vendor lookup for common manufacturers

Data persistence

Chats, messages, peer profiles, and metadata are serialized to JSON and stored in app-private storage, with photos as JPEG files referenced by URI.

  • Automatic save on background, ViewModel clear, and every 5 minutes
  • Automatic load on app start
  • Graceful handling of missing/new fields and corrupt entries

Background services

The background service keeps BLE advertising and periodic location updates active when the app is minimized, with a persistent, low-priority notification.

  • Broadcasts your location every 10 seconds in background
  • Requires foreground & background location permissions
  • Automatically restarts to recover after process death
07 · Architecture

MVVM, repositories & a shared ViewModel core

Radar is built on a clean MVVM architecture with a central shared ViewModel that owns device collections, chats, profile, scanners, and message delivery services.

Core patterns

  • MVVM with LiveData and lifecycle-aware observers
  • Repository layer (data persistence, profile repository, etc.)
  • Service layer for scanners, messaging, and location
  • Transport-agnostic abstraction for message delivery

Key components

  • Shared ViewModel – central state & collections
  • Scanners – Bluetooth, WiFi Aware, mDNS, Network Segment
  • Services – message delivery, foreground service, location pusher, peers fetcher
  • Data collections – chats, peers, Bluetooth devices, WiFi devices, all devices
08 · Special Features

Drone detection, SMS integration, FCM & aircraft tracking

Beyond core messaging and presence, Radar includes airspace awareness, SMS integration, and cloud-assisted notifications via Firebase Cloud Messaging.

Drone detection (Remote ID)

Decodes ASTM F3411-22 and ASD-STAN prEN 4709-002 Remote ID messages over BLE, showing drone ID, GPS coordinates, altitude, speed, heading, and operational status.

Dedicated Remote ID Bluetooth service UUID Basic ID, Location & Message Pack

SMS integration

Data model and SMS inbox reader support importing SMS messages, normalizing phone numbers, grouping by thread, and mapping to virtual profiles with deterministic UUIDs.

Firebase Cloud Messaging

The Firebase messaging service registers an FCM token with the Oracle backend so that message relays can trigger push notifications that deep-link to the correct chat.

Aircraft tracking (OpenSky)

Optional connection to the OpenSky Network API to render real-time aircraft positions on the map view with heading-aligned airplane icons and callsigns.

10 · Use Cases

Where Radar shines

Event networking

Conferences, concerts, and festivals with 100+ attendees where cellular networks are congested – discover nearby people and chat locally.

Emergency communication

Natural disasters or outages where infrastructure is down. Use WiFi Aware and BLE for local messaging between nearby devices.

Outdoor activities

Hiking, camping, or skiing trips where you want to keep track of your group’s positions without relying on mobile coverage.

Large venue coordination

Stadiums, arenas, and convention centers where friends are spread out and cellular is overloaded – Radar helps coordinate meeting points.

Network administration

Office and home network audits to discover devices, identify vendors, and spot unknown or rogue hosts on the LAN.

Drone & airspace awareness

Security teams can detect and track Remote ID-broadcasting drones around a site, alongside aircraft overlays via the OpenSky Network.

  • Mesh store-and-forward message relay
  • Visual radar & map views with real-time positioning
  • Profile exchange with automatic photo transmission
